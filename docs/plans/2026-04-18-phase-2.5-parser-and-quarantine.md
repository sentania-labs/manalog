# Phase 2.5 Plan — MTGO Log Parser + Quarantine Workflow

Builds on `docs/research/2026-04-18-mtgo-log-structure.md`. Refines after empirical analysis of 358 real `Match_GameLog_*.dat` samples.

---

## What changed vs the research brief

Three corrections from the brief, plus one new design layer:

1. **`.dat` is parseable directly — no BigPeet bridge needed.** The files contain readable UTF-8 text fragments separated by binary framing. `bytes.decode("utf-8", errors="replace")` followed by pattern matching works. The brief's recommendation to subprocess-call `BigPeet/mtgo_utils file_parser.exe` is dropped. Removes a vendored binary, an external dependency, and a Windows-only code path.

2. **`wins the match X-Y` *does* appear in real logs** — once per completed match. The brief's claim that this string never appears (and that the current parser will therefore never fire) was wrong. The actual bugs in the current parser are different and worse:
   - Whose-perspective blindness: parser captures any `"X wins the match"` and reports `result="win"` regardless of whether `X` is the user (5 of 12 sample losses currently mis-reported as wins).
   - `LOSES_PATTERN` is dead code — the string `"loses the match"` never appears.
   - `OPPONENT_PATTERN` matches the literal word "opponent" in normal play text, capturing `"draws"`, `"creates"`, `"reveals"` as opponent names.
   - `mtgo_match_id` is non-stable (UUID-per-parse) — confirmed.

3. **Loss-side match terminator is `<P> has run out of time and has lost the match.`** Found in 4 / 358 samples. The brief's expected `"loses the match"` doesn't exist standalone; it only appears inside this longer phrase. This is the only authoritative loss-side signal — it can fire **before** a player reaches 2 game wins (replacing `wins the match X-Y` entirely on a time loss).

4. **Dead-letter quarantine.** New design layer not in the brief. Files the parser can't confidently bucket are uploaded raw to the server for inspection and reprocessing. Closes the loop on undocumented format edge cases (BO1, special win conditions, draws, future MTGO format drift) — instead of guessing what they look like, we capture them in production and update the parser when we see them.

## Architecture

### Shared parser module

Move `agent/parser.py` to a shared location both agent and server import:
- `app/services/log_parser.py` (server-side canonical home)
- Agent imports the same module via packaging (PyInstaller bundles it)

Agent ships its pinned version; server runs latest. Reprocessing of quarantined logs uses the server-side parser — so when Scott updates parser logic, redeploys, and triggers reparse, every quarantined log gets a fresh attempt.

### Local agent state (SQLite)

`%APPDATA%\MTGOMatchTracker\state.db`:

```
submissions (
  mtgo_match_id   TEXT PRIMARY KEY,
  parsed_at       TIMESTAMP,
  payload_kind    TEXT,           -- 'parsed' | 'quarantined'
  submitted_at    TIMESTAMP,
  server_ack      BOOLEAN,
  last_error      TEXT,
  retry_count     INT
)
```

Survives restart; prevents re-uploads when the watcher re-fires; powers a retry queue for failed sends. Server-side dedup on `mtgo_match_id` is the safety net, not the primary defense.

### Server-side schema additions

**Add column** to `matches`:
- `raw_log_text TEXT NULL` — cleaned text dump from the parsed `.dat` (~10–40KB per match). Lets us re-derive `games`/`plays` from the canonical text without needing the user to re-export logs, if the parser improves.

**New table** `unparsed_logs`:

```sql
unparsed_logs (
  id              SERIAL PRIMARY KEY,
  user_id         INT NOT NULL REFERENCES users(id),
  agent_id        UUID REFERENCES agent_registrations(agent_id),
  mtgo_match_id   TEXT NOT NULL,
  raw_log_text    TEXT NOT NULL,
  failure_reason  TEXT NOT NULL,         -- enum below
  parser_version  TEXT NOT NULL,
  status          TEXT NOT NULL,         -- 'pending' | 'parsed' | 'unparseable'
  retry_count     INT DEFAULT 0,
  last_retry_at   TIMESTAMP,
  created_at      TIMESTAMP DEFAULT NOW(),
  UNIQUE (user_id, mtgo_match_id)
)
```

`matches.mtgo_match_id` and `unparsed_logs.mtgo_match_id` share the same value space (filename GUID). Each `(user_id, mtgo_match_id)` should appear in exactly one of the two tables. Successful reparse migrates a row from `unparsed_logs` (status `parsed`) into `matches` + `games` + `plays`; the `unparsed_logs` row is retained as audit + regression-test data.

### New endpoints

| Method | Path | Purpose |
|---|---|---|
| `POST /api/v1/agent/upload` | (existing) | Confidently parsed match |
| `POST /api/v1/agent/upload-unparsed` | NEW | Raw text + failure reason |
| `GET /api/v1/admin/unparsed` | NEW | List quarantined logs (filter by `failure_reason`, `status`) |
| `GET /api/v1/admin/unparsed/{id}` | NEW | Inspect one (returns `raw_log_text`) |
| `POST /api/v1/admin/unparsed/reparse` | NEW | Trigger re-parse over all/filtered quarantined logs |

Admin endpoints gated by user role (Scott as superuser for MVP; multi-user role model is a later concern).

## Parser rule set (consolidated from 358-sample analysis)

All patterns matched against `bytes.decode("utf-8", errors="replace")` of the `.dat` file. Username capture uses `(.+?)` non-greedy stopping at the next control byte or `@` to handle `Djas Puhr`, `Bosh N Roll` and any future space-containing usernames. Score capture uses `(\d-\d)` (single digits) to avoid the `2-11` greed bug.

| Signal | Meaning |
|---|---|
| `@P(.+?) rolled a (\d+)` | Match start; both players appear once each (4× in re-rolls) |
| `@P(.+?) joined the game` | Player roster — opponent = roster minus self (self from agent config) |
| `@P(.+?) chooses to (not )?play first` | On-play / on-draw |
| `@P(.+?) begins the game with (\w+) cards in hand` | Starting hand size; word-to-int from cderickson's table |
| `@P(.+?) mulligans to (\w+) cards?` | Mulligan event |
| `Turn (\d+):` | Turn boundary |
| `@P(.+?) wins the game` | Game winner |
| `@P(.+?) has conceded from the game` | Game loser (concession) |
| `@P(.+?) has lost connection to the game` | Game loser (disconnect) |
| `@P(.+?) has left the game` | Game loser (closed client) |
| `Match Tied (\d-\d)` | Decorative game-2-boundary marker; ignore |
| `@P(.+?) wins the match (\d-\d)` | Match winner (BO3 normal completion) |
| `@P(.+?) leads the match (\d-\d)` | In-progress; **skip the file entirely** |
| `@P(.+?) has run out of time and has lost the match` | Match loser (time loss) — overrides game tally |

**Per-play extraction** (cards), inside any action line:
- Card reference: `@\[(.+?)@:(\d+),(\d+):@\]` → `(name, card_id, instance_id)`
- Action verbs (taxonomy): `plays @\[` → land, `casts @\[` → cast, `activates an ability of` → ability, `puts triggered ability from` → trigger, `draws a card with` → draw, `discards @\[` → discard, `is being attacked by` → attack, `sacrifices @\[` → sacrifice

## Confidence triage — which path does the agent take?

The agent inspects parser output and picks one of three paths per file:

### → `POST /agent/upload` (confident parse)

All of:
- Exactly one match-end signal present (`wins the match X-Y` *or* `has run out of time and has lost the match`)
- ≥1 game with an extracted winner
- Both player names extracted, including self
- Game count ≥ winning side's game tally (`2-0` → ≥2 games; `2-1` → ≥3)

### → `POST /agent/upload-unparsed` (quarantine)

Any of:
- `no_winner` — no `wins the game` line at all (and not in-progress)
- `unknown_match_end` — game wins present but no `wins the match` / `has lost the match` line
- `score_mismatch` — extracted game count doesn't match score line
- `empty_player` — couldn't extract opponent name
- `low_confidence` — catch-all for parser-flagged anomalies

The raw text payload is the cleaned UTF-8 dump, not the binary `.dat`.

### → Skip locally (no server contact)

- File contains `leads the match X-Y` with no `wins the match X-Y` follow-up — in-progress; retry on next file event
- File contains no recognizable signals at all (e.g. the 272-byte aborted file before game 1)

The local state DB records skipped files (with `payload_kind='skipped-in-progress'`) so the watcher can reuse the decision on re-fire.

## Reprocessing flow

1. Scott opens `/admin/unparsed` (initially via API client; UI later)
2. Filters by `failure_reason`, picks a representative case
3. Fetches `raw_log_text` via `/admin/unparsed/{id}`
4. Updates parser logic in `app/services/log_parser.py`, ships
5. Calls `POST /admin/unparsed/reparse` (optionally filtered)
6. Server iterates `unparsed_logs WHERE status='pending'`, runs current parser over each `raw_log_text`:
   - Success → write `matches`/`games`/`plays`, mark `unparsed_logs.status='parsed'`, retain row
   - Failure → bump `retry_count`, update `last_retry_at`, leave `status='pending'`
7. After several deploys without progress on a row, manually mark `status='unparseable'` to silence it from the inbox

`unparsed_logs` rows are never deleted — they double as parser regression test fixtures.

## Implementation order

| Step | What | Side |
|---|---|---|
| 1 | Move parser to `app/services/log_parser.py`; agent imports the same module | Both |
| 2 | Rewrite parser with the consolidated rule set; filename-GUID `mtgo_match_id`; self-username from agent config to disambiguate winner perspective | Shared |
| 3 | Alembic migration: `matches.raw_log_text` column + `unparsed_logs` table | Server |
| 4 | `POST /agent/upload-unparsed` endpoint + Pydantic schema | Server |
| 5 | Admin endpoints: list, get, reparse | Server |
| 6 | Local agent state SQLite (submissions tracking + retry) | Agent |
| 7 | Update agent upload flow: pick endpoint by confidence; consult state DB to dedup | Agent |
| 8 | Path discovery fix: glob `%LOCALAPPDATA%\Apps\2.0\` for `Match_GameLog_*.dat` rather than hardcoded path | Agent |
| 9 | Anonymized fixture set (subset of the 358 samples with usernames hashed) under `tests/fixtures/mtgo_logs/` | Tests |
| 10 | Parser unit tests over fixtures + quarantine flow integration test | Tests |

Steps 1–2 unblock everything else. Steps 3–5 enable the quarantine loop. Steps 6–8 round out agent reliability.

## Out of scope for Phase 2.5

- BigPeet `file_parser` bridge (dropped — direct text extraction works)
- DraftLog tracking (separate directory and format)
- Decklist extraction (logs don't contain decklists)
- Format / match-type detection from log content (must come from agent config — log mentions of "legacy"/"league" are deck names or card text, not authoritative)
- Auto-classification of archetype (separate workstream)
- Multi-user admin role model (single superuser for MVP)

---

## Test plan

- **Parser unit tests** over the anonymized fixture set. Per-file expected match outcome, game count, on-play, mulligans, player extraction. Includes the known oddballs: in-progress, aborted, time loss, space-username, mid-snapshot.
- **Quarantine flow integration test**: feed a deliberately broken log, assert it lands in `unparsed_logs` (not `matches`). Update parser stub to handle it, run reparse, assert successful migration.
- **Agent state test**: simulate restart with mid-batch failures; assert no duplicate submissions on resume.
- **Coverage gate** unchanged at 70%; `app/services/log_parser.py` should comfortably exceed that given the fixture set.

## Known unknowns the corpus didn't surface

Documented here so they're recognized when they arrive (likely as quarantine entries, exactly as designed):

- BO1 matches (Standard / Pauper Leagues — 100% of the 358 corpus is BO3)
- Special win conditions (Thassa's Oracle, Approach of the Second Sun, mill, poison/infect)
- Match draws
- Unicode-character usernames (international players)
- MTGO format-drift events (last known break: July 2019)
